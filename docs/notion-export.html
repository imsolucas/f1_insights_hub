1. Project Overview

Project Name: F1 Insight Hub

Description: A full-stack web application for Formula 1 fans to explore race data, statistics, predictions, and AI-generated insights. Built as a portfolio project demonstrating full-stack development, ML integration, and modern web technologies.

Core Goals

  • Provide comprehensive F1 race data and statistics
  • Enable race outcome predictions using machine learning
  • Generate AI-powered race summaries and insights
  • Create an engaging, data-first user experience
  • Demonstrate full-stack + ML integration capabilities

Core Features by Phase

Phase 1 - MVP (Current)

  • Race schedule and results browsing
  • Driver and team statistics
  • Circuit information
  • Basic data visualization

Phase 2 - ML Integration

  • Race outcome prediction models
  • Driver performance trend analysis
  • ML-powered insights

Phase 3 - AI Features

  • AI-generated race summaries
  • F1 chatbot for questions
  • Personalized insights

Phase 4 - Polishing

  • Advanced charts and visualizations
  • User authentication and profiles
  • Leaderboards and comparisons
  • Performance optimizations

Tech Stack Summary

  • Frontend: Next.js 16, React 19, Tailwind CSS 4, shadcn/ui, TanStack Query
  • Backend: Node.js, Express, TypeScript, Prisma ORM
  • Database: PostgreSQL (Render free tier)
  • ML: Python 3.10+, FastAPI, scikit-learn (Phase 2+)
  • AI: OpenAI API (Phase 3+)
  • Deployment: Vercel (frontend), Render (backend + DB), GitHub Actions (CI/CD)

Component Breakdown

Backend (Express)

  • Controllers: Request handlers for each resource
  • Services: Business logic layer
  • Repositories: Data access layer (Prisma)

Database (PostgreSQL)

  • Schema: Prisma-managed models
  • Tables: drivers, constructors, circuits, races, results, qualifying
  • Indexes: Optimized for common queries
  • Migrations: Version-controlled schema changes

  • Endpoints: Prediction endpoints

4. Database Schema Design

Tables

Table Purpose Key Fields
drivers Driver information id, driverId, forename, surname, nationality, permanentNumber
constructors Team/constructor information id, constructorId, name, nationality
circuits Circuit/track information id, circuitId, name, location, country, lat, long
seasons Season years id, year
races Race information id, season, round, raceName, circuitId, date, time
race_results Race finishing positions and points id, raceId, driverId, constructorId, position, points, status
qualifying_results Qualifying session results id, raceId, driverId, constructorId, position, q1, q2, q3

Relationships

  • Race ‚Üí Circuit (many-to-one)
  • Race ‚Üí Season (many-to-one)
  • RaceResult ‚Üí Race (many-to-one)
  • RaceResult ‚Üí Driver (many-to-one)
  • RaceResult ‚Üí Constructor (many-to-one)
  • QualifyingResult ‚Üí Race (many-to-one)
  • QualifyingResult ‚Üí Driver (many-to-one)
  • QualifyingResult ‚Üí Constructor (many-to-one)

Indexes

  • races: season, circuitId, date
  • race_results: raceId, driverId, constructorId
  • qualifying_results: raceId, driverId
  • drivers: driverId (unique)
  • constructors: constructorId (unique)
  • circuits: circuitId (unique)

Data Synchronization Strategy

  1. On server startup: Sync current season schedule and drivers/constructors
  2. Periodic sync: Check for new race results (after race weekends)
  3. Cache-first: Check database before calling Ergast API
  4. Manual trigger: Admin endpoint to force sync
  5. Historical data: Fetch on-demand when user requests specific seasons

7. API Documentation

Phase 1 Endpoints

Races

  • GET /api/v1/races - List all races (with pagination, filtering by season)
  • GET /api/v1/races/:raceId - Get race details
  • GET /api/v1/races/:raceId/results - Get race results
  • GET /api/v1/races/:raceId/qualifying - Get qualifying results
  • GET /api/v1/races/current/schedule - Get current season schedule

Drivers

  • GET /api/v1/drivers - List all drivers (with pagination, filtering by season)
  • GET /api/v1/drivers/:driverId - Get driver details and stats
  • GET /api/v1/drivers/:driverId/results - Get driver race results history

Constructors/Teams

  • GET /api/v1/constructors - List all constructors (with pagination, filtering)
  • GET /api/v1/constructors/:constructorId - Get constructor details and stats
  • GET /api/v1/constructors/:constructorId/results - Get constructor results history

Circuits

  • GET /api/v1/circuits - List all circuits
  • GET /api/v1/circuits/:circuitId - Get circuit details
  • GET /api/v1/circuits/:circuitId/results - Get all races at this circuit

Response Format

{
  "success": true,
  "data": { ... },
  "correlationId": "uuid",
  "timestamp": "ISO-8601"
}

Error Response Format

{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "Human-readable error message",
    "details": { ... }
  },
  "correlationId": "uuid",
  "timestamp": "ISO-8601"
}

8. Development Workflow

Branch Strategy

  • main: Production branch, triggers deployment
  • develop: Staging branch (optional), can trigger staging deployment
  • feature/*: Feature branches, run verification only

Commit Conventions

Use conventional commits:

  • feat: New feature
  • fix: Bug fix
  • docs: Documentation changes
  • style: Code style changes (formatting, etc.)
  • refactor: Code refactoring
  • test: Test additions or changes
  • chore: Build process or auxiliary tool changes

CI/CD Pipeline

  1. Push to repository triggers GitHub Actions
  2. Verify job runs: typecheck, lint, build, test
  3. If on main branch and verify succeeds:
    • Deploy backend to Render
    • Run database migrations
    • Deploy frontend to Vercel

Pre-deployment Checks

  • TypeScript type checking: pnpm typecheck
  • Linting: pnpm lint
  • Build verification: pnpm build
  • Migration status: npx prisma migrate status

9. Cost Management

Free Tier Limits

Service Free Tier Limit Notes
Vercel Hobby plan Unlimited for personal projects
Render Free tier Spins down after inactivity, 750 hours/month
Render PostgreSQL Free tier 90 days retention, limited connections
Ergast API Free 4 requests/second rate limit
OpenAI API Pay-as-you-go Set usage caps, monitor token usage

Optimization Strategies

  • Cache Ergast API responses aggressively (historical data doesn't change)
  • Use database indexes for faster queries
  • Implement pagination to reduce payload sizes
  • Cache AI responses to reduce OpenAI API calls
  • Monitor API usage and set alerts
  • Use appropriate OpenAI models (GPT-3.5-turbo for most tasks)
